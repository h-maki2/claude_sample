# プロジェクトメモ

## 必須ルール

### 1. スキルの呼び出し

以下の作業では、対応するスキルを**必ず最初に呼び出してから**作業を開始すること。

| 作業内容 | 必須スキル |
|---------|-----------|
| テストコードを作成・修正する | `test-writing` |
| TDDサイクルで実装する | `tdd` |
| プロダクションコードを実装・修正する | `development-guidelines` |
| 作業計画を立てる・実装する・振り返る | `steering` |
| ER図からマイグレーションを作成する | `laravel-migrate` |
| 新しいモジュールを作成する | `laravel-module` |

「修正だから不要」「小さいから不要」という判断は禁止。

### 2. テスト実行

テストは必ず `test-runner` サブエージェント経由で実行する。

**モード選択:**
- ユーザーが「全てのテスト」「統合テスト」「featureテスト」と明示 → **モード2（全テスト）**
- プロダクションコード変更後の自動確認 → **モード1（単体テストのみ）**

プロンプトには対象・モード・作業ディレクトリ `/var/www/html` を明示する。テストコマンドは含めない（test-runner が自律的に決定する）。

## プロジェクト構造

```
docs/
├── <サブドメイン英語名>/   # 永続ドキュメント（PRD・ドメインモデル図・ER図など）
├── steering/              # 作業単位ドキュメント（詳細は steering スキル参照）
└── ubiquitousLanguage/
    └── ubiquitous_language.csv  # 全サブドメイン共通のユビキタス言語集
src/                       # Laravelアプリケーション本体
```

**現在のサブドメイン:** `ReservationManagement`（予約管理）、`MeetingRoomManagement`（会議室管理）

実装時はユビキタス言語 CSV を参照して命名を統一すること。

---

## 新規開発フロー

新しいサブドメインを開発する際はスラッシュコマンドを使う:

```
/create-docs <サブドメイン名>      # ドキュメント一式（PRD・図）を生成
/implement <サブドメイン名> <内容>  # ステアリング作成 → TDD実装
```
