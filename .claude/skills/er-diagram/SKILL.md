# ER図生成スキル

このスキルは、ドメインモデル図（`domain-model.mmd`）をもとに、**リレーショナルDB設計の正規化原則**に従ったER図をMermaid記法で自動生成します。

> **重要**: ドメインモデル図はビジネス概念の整理であり、テーブル設計とは別物です。ドメインモデルの構造をそのままテーブルに写すのではなく、正規化の観点から改めてテーブル構造を設計してください。

## 起動時の共通処理

### 1. サブドメイン名の解決

`resolve-subdomain` スキルを呼び出してビジネスサブドメインの英語名を取得します:

```
Skill("resolve-subdomain", "$1")
```

取得した `<英語名>` を以降のすべてのファイルパスで使用します。

### 2. モードの自動判別

`docs/<英語名>/ER.mmd` の存在を確認し、実行するモードを決定します。

- **ファイルが存在しない** → モード1（新規作成）を実行する
- **ファイルが存在する** → モード2（更新）を実行する

---

## モード1: ER図新規作成

### 目的

ドメインモデル図（`domain-model.mmd`）をもとに、ER図を**新規作成**します。

### 入力

- **ビジネスサブドメイン名**: 引数として渡される（例: `予約管理`、`会議室管理`）
- **ドメインモデル図ファイル**: `docs/<英語名>/domain-model.mmd`

### 出力先

```
docs/<英語名>/ER.mmd
```

### 実行手順

#### 1. ドメインモデル図ファイルの読み込み

以下のパスにあるドメインモデル図ファイルを読み込みます:

```
docs/<英語名>/domain-model.mmd
```

ファイルが存在しない場合は、ユーザーに以下のメッセージを伝えてください:

```
docs/<英語名>/domain-model.mmd が見つかりません。
domain-modelスキルでドメインモデル図を先に作成してください。
```

#### 2. テーブル設計（正規化）

ドメインモデル図を参照しながら、以下の手順でテーブルを設計します。ドメインモデルの構造は**ビジネス概念の理解**に使い、テーブル設計はリレーショナルDB設計の正規化原則に従って独立して行います。

#### Step 1: 管理すべきデータの洗い出し

ドメインモデル図を参照して、システムが永続化すべきデータを列挙します。

- ドメインモデル図に登場する概念のうち、データとして保存が必要なものは何か
- アクターそのものをデータとして管理する必要があるか

#### Step 2: 第1正規形（1NF）の適用

以下のルールに従い、データをテーブルの形に整理します:

- 1つのセルには1つの値のみ（繰り返しグループや配列は別テーブルへ）
- 各行を一意に識別できる主キーを設ける
- 例: 「予約が複数の参加者を持つ」→ `予約参加者` テーブルを別途作成する

#### Step 3: 第2正規形（2NF）の適用

複合主キーのテーブルで、非キー属性が主キーの一部だけに依存していないか確認します:

- 複合主キーの片方のキーだけで決まる属性は、別テーブルに分離する
- 例: `(予約ID, 会議室ID)` が主キーのテーブルで「会議室名」が会議室IDだけで決まる → `会議室` テーブルに移動する

#### Step 4: 第3正規形（3NF）の適用

非キー属性が他の非キー属性に依存していないか確認します（推移的関数従属の排除）:

- 非キー属性Aが非キー属性Bを決定する場合は、別テーブルに分離する
- 例: `予約テーブル` に「会議室名」「会議室の収容人数」があり、どちらも「会議室ID」で決まる → `会議室` テーブルに分離する

#### Step 5: 多対多関係の中間テーブル化

以下の場合は中間テーブルを作成します:

- エンティティAとエンティティBが多対多の関係にある場合
- 例: 「1つの予約に複数のユーザーが参加でき、1人のユーザーが複数の予約に参加できる」→ `予約参加者` 中間テーブルを作成する
- 中間テーブルの主キーは通常、両方のエンティティのIDの複合キーとする

#### Step 6: 外部キーの定義

テーブル間の参照関係を外部キーとして定義します:

- 「1対多」の関係では、「多」側のテーブルに「1」側のIDを外部キーとして追加する
- 中間テーブルには両方のエンティティのIDを外部キーとして持たせる

#### Step 7: カラムとデータ型の確定

各テーブルのカラムを確定し、適切なデータ型を割り当てます:

| データの種類 | 型 |
|---|---|
| 識別子（ID） | `string` または `int` |
| 名称・テキスト | `string` |
| 整数値・数量 | `int` |
| 金額・小数 | `decimal` |
| 日時 | `datetime` |
| 日付のみ | `date` |
| 真偽値・フラグ | `boolean` |
| 区分・ステータス | `string`（enumの値を文字列として保存） |

#### 3. MermaidER図の生成

以下のフォーマットに従ってMermaidの `erDiagram` でER図を生成します。

```
%% ER図: <ビジネスサブドメイン名>
erDiagram

    TABLE_NAME {
        type column_name PK "説明"
        type column_name FK "外部キー: 参照先テーブル名"
        type column_name "説明"
    }

    TABLE_A ||--o{ TABLE_B : "関連ラベル"
```

#### 命名規則

- **テーブル名**: ドメインモデル図に登場する概念の英語名（スネークケース大文字）を使用（例: `RESERVATION`, `MEETING_ROOM`）
- **カラム名**: ドメインモデル図の属性から読み取れる項目の英語名（スネークケース小文字）を使用（例: `reservation_id`, `started_at`）
- 英語名が判断できない場合は日本語名をそのまま使用する

#### キー制約の表記

- `PK` : 主キー
- `FK` : 外部キー（コメントに参照先テーブル名を記載）
- `PK, FK` : 主キー兼外部キー（中間テーブル等）

#### カーディナリティの記法

| 記法 | 意味 |
|---|---|
| `\|\|--\|\|` | 1対1（必須） |
| `\|\|--o\|` | 1対0〜1 |
| `\|\|--o{` | 1対多（0以上） |
| `\|\|--\|{` | 1対多（1以上） |
| `}o--o{` | 多対多（中間テーブルで表現すること） |

#### 生成例

```
%% ER図: 予約管理
erDiagram

    MEETING_ROOM {
        string meeting_room_id PK "主キー"
        string name "会議室名"
        int capacity "収容人数"
        boolean is_active "利用可否"
    }

    RESERVATION {
        string reservation_id PK "主キー"
        string meeting_room_id FK "外部キー: MEETING_ROOM"
        string name "予約名"
        string contact_person "担当者名"
        string email "連絡先メール"
        datetime started_at "開始日時"
        datetime ended_at "終了日時"
        string status "ステータス"
    }

    MEETING_ROOM ||--o{ RESERVATION : "has"
```

#### 4. ファイルの保存

出力先ディレクトリが存在しない場合は作成し、生成したMermaidコードを以下のパスに保存します:

```
docs/<英語名>/ER.mmd
```

保存後、ユーザーに以下のメッセージを伝えてください:

```
ER図を docs/<英語名>/ER.mmd に保存しました。

定義したテーブル:
- [テーブル名一覧]

Mermaid対応のエディタ（VSCode + Mermaid拡張、Mermaid Live Editorなど）でプレビューできます。
```

### 使い方・実行イメージ

Claude Codeを起動し、以下のようにビジネスサブドメイン名を渡して話しかけるだけです。

**入力例:**
```
/er-diagram 予約管理
/er-diagram 会議室管理
/er-diagram 決済
```

## モード2: ER図の更新

### 目的

既存のER図（`ER.mmd`）とドメインモデル図（`domain-model.mmd`）を照合し、ER図を最新の状態に更新します。既存の記述も含めて自由に変更・追記できます。

### 入力

- **ビジネスサブドメイン名**: 引数として渡される（例: `予約管理`、`会議室管理`）
- **既存ER図**: `docs/<英語名>/ER.mmd`
- **ドメインモデル図**: `docs/<英語名>/domain-model.mmd`

### 出力先

```
docs/<英語名>/ER.mmd（上書き更新）
```

### 実行手順

#### 1. 既存ER図の読み込み

`docs/<英語名>/ER.mmd` を読み込みます。

#### 2. ドメインモデル図の読み込み

`docs/<英語名>/domain-model.mmd` を読み込みます。

ファイルが存在しない場合は、ユーザーに以下のメッセージを伝えてください:

```
docs/<英語名>/domain-model.mmd が見つかりません。
domain-modelスキルでドメインモデル図を先に作成してください。
```

#### 3. 差分の特定

ドメインモデル図と既存ER図を照合し、**ER図に未反映の要素**を特定します。

**テーブルの未反映判定**:
- ドメインモデル図に登場するエンティティ・集約ルートのうち、ER図にテーブルとして存在しない → **未反映**
- すでにテーブルとして記載されている → **反映済み（スキップ）**

**カラムの未反映判定**:
- ドメインモデル図のエンティティ・値オブジェクトの属性のうち、対応テーブルのカラムとして存在しない → **未反映**

**リレーションの未反映判定**:
- ドメインモデル図の関連（`*--`, `-->`）のうち、ER図のリレーション行として存在しない → **未反映**

未反映の要素がひとつもない場合は、更新を行わずユーザーに以下のメッセージを伝えてください:

```
ドメインモデル図の内容はすべてER図に反映されています。更新は不要です。
```

#### 4. 追記内容の決定

未反映の要素について、モード1の「テーブル設計（正規化）」の手順（Step 1〜7）に従い、追記するテーブル・カラム・リレーションを決定します。

#### 5. ファイルの更新

モード1の「テーブル設計（正規化）」「MermaidER図の生成」ルールに従い、既存の内容と追記・変更内容をまとめてER図を再生成し、ファイルを上書き保存します。既存の記述も必要に応じて自由に変更・削除できます。

保存後、ユーザーに以下のメッセージを伝えてください:

```
ER図を更新しました: docs/<英語名>/ER.mmd

【変更内容】
- 追記したテーブル: <追記したテーブル名一覧（なければ省略）>
- 変更したテーブル: <変更したテーブル名一覧（なければ省略）>
- リレーション: <追記・変更したリレーション一覧（なければ省略）>
```

### 使い方・実行イメージ

モード1・モード2ともに入力形式は同じです。ファイルの存在によって自動的にモードが判別されます。

**入力例:**
```
/er-diagram 予約管理
/er-diagram 会議室管理
```
