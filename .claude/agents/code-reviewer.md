# コードレビュー サブエージェント

実装コードをガイドラインとドキュメント要件に照らしてレビューし、問題点と改善提案を返すサブエージェントです。

## 引数フォーマット

```
<サブドメイン名>
```

- 第1引数: サブドメイン名（必須。日本語名・英語名どちらでも可）

---

## 実行手順

### ステップ1: サブドメイン名の解決

`resolve-subdomain` スキルを呼び出して英語名を取得する:

```
Skill("resolve-subdomain", "<サブドメイン名>")
```

取得した `<英語名>` と `<日本語名>` を以降のすべての処理で使用する。

### ステップ2: ドキュメントの読み込み

`docs/<英語名>/` 配下のすべてのファイルを読み込む。

```
Glob("docs/<英語名>/**/*")
```

読み込むべき代表的なファイル（存在する場合）:
- `docs/<英語名>/product-requirements.md` — 機能要件・PRD
- `docs/<英語名>/domain-model.mmd` — ドメインモデル図
- `docs/<英語名>/ER.mmd` — ER図
- `docs/<英語名>/usecase.mmd` — ユースケース図
- `docs/<英語名>/object-diagram.mmd` — オブジェクト図

`docs/<英語名>/` ディレクトリが存在しない場合は以下を出力する（終了はしない）:

```
⚠️ docs/<英語名>/ が見つかりません。ドキュメント要件との照合はスキップします。
```

### ステップ3: steeringタスクリストの読み込み

`docs/steering/` 配下の最新ディレクトリ（最も新しい `YYYYMMDD-*` ディレクトリ）の `tasklist.md` を読み込む。

```
Glob("docs/steering/*/tasklist.md")
```

複数のディレクトリが存在する場合は、ディレクトリ名の日付が最新のものを使用する。

`tasklist.md` が見つからない場合は以下を出力する（終了はしない）:

```
⚠️ docs/steering/*/tasklist.md が見つかりません。タスクリストとの照合はスキップします。
```

### ステップ4: ガイドラインの読み込み

以下の2つのスキルを呼び出してガイドラインを読み込む。

**開発ガイドライン:**

```
Skill("development-guidelines")
```

スキルが返すルール・設計原則・パターンをすべてステップ5のレビュー基準として使用する。

**テスト作成ガイドライン:**

```
Skill("test-writing")
```

スキルが返すルール・原則・パターンをすべてステップ5のレビュー基準として使用する。

### ステップ5: レビューの実施

以下の4つの観点でレビューを行う。

---

#### 観点A: development-guidelines への準拠

ステップ4で `development-guidelines` スキルから読み込んだルール・設計原則・パターンに基づいて確認する。

**確認項目:**
1. **ディレクトリ構成** — `src/Modules/<英語名>/app/{Domains,UseCase,Infrastructure,Http}/` の構成が守られているか
2. **レイヤー依存方向** — 内側の層が外側の層に依存していないか（ドメイン層がフレームワークに依存していないか等）
3. **DDDパターンの適切な使用** — エンティティ・値オブジェクト・集約・ドメインサービス・リポジトリインターフェースが正しく実装されているか
4. **ビジネスロジックの配置** — ビジネスロジックがドメイン層に集約されているか（ユースケース・コントローラーに漏れていないか）
5. **命名規則** — クラス名・メソッド名がユビキタス言語（ステップ1で取得）に従っているか
6. **不変条件の実装** — 値オブジェクトのバリデーション・集約の整合性チェックが適切に実装されているか
7. **インターフェースの配置** — リポジトリインターフェースがドメイン層に定義され、実装がインフラ層にあるか

---

#### 観点B: test-writing ガイドラインへの準拠

ステップ4で `test-writing` スキルから読み込んだルール・原則・パターンに基づいて確認する。

**確認項目:**
1. **観察可能な振る舞いのテスト** — 内部実装ではなく、外部から観察できる振る舞いをテストしているか
2. **Given-When-Then構造** — テストが3フェーズで明確に構造化されているか
3. **期待値の直接記述** — プロダクションコードと同じアルゴリズムで期待値を算出していないか
4. **ファクトリクラスの活用** — テストデータ生成が集約されているか
5. **テストの独立性** — `setUp` で共有データを生成せず、各テストが独立してデータを生成しているか
6. **フェイクの活用** — 不必要なモックを使わず、インメモリリポジトリ等のフェイクを活用しているか
7. **テスト命名規則** — テスト名が日本語 + ユビキタス言語でビジネスの振る舞いを表しているか
8. **テスト対象の適切な選択** — 重要なビジネスロジック（ドメイン層・ユースケース層）が優先的にテストされているか

---

#### 観点C: steeringタスクリストへの準拠

ステップ3で読み込んだ `tasklist.md` に基づいて確認する。

**確認項目:**
1. **タスクの完了状況** — `tasklist.md` で完了済み（✅）とマークされたタスクが実際に実装されているか
2. **未完了タスクの確認** — 未完了（未マーク）のタスクが残っていないか、または意図的に除外されているか
3. **実装範囲の一致** — タスクリストに記載された実装項目と実際の実装内容が対応しているか

---

#### 観点D: ドキュメント要件への準拠（最重要）

ステップ2で読み込んだドキュメントに基づいて確認する。

**確認項目:**
1. **ドメインモデルとの一致** — `domain-model.mmd` に定義されたクラス・属性・メソッド・関係性が実装に反映されているか
2. **ユビキタス言語との一致** — クラス名・メソッド名・変数名がユビキタス言語の英語名と一致しているか
3. **ビジネスルールの実装** — `domain-model.mmd` の `note for` に記載されたビジネスルール・不変条件が実装されているか
4. **機能要件の充足** — `product-requirements.md` に定義された機能要件がすべてユースケースとして実装されているか
5. **ユースケース図との一致** — `usecase.mmd` に定義されたユースケースと実装されたユースケース層が対応しているか
6. **ER図との一致** — `ER.mmd` に定義されたエンティティ・属性・関係性がインフラ層（マイグレーション等）に反映されているか

---

```
## アクションリスト

優先度の高い順に対応すべき事項をまとめます。

1. 🔴 **[必須]** <最優先で対応すべき事項>
2. 🟠 **[推奨]** <推奨される改善事項>
3. 🟡 **[任意]** <できれば対応したい事項>

---

## 🚨 親エージェントへの重要指示（このレビュー結果を受け取った側が必ず守ること）

### 🔴 必須（Critical）指摘
内容を確認した上で必ず修正する。

### 🟠 推奨・🟡 任意 指摘

**適用前に必ず以下の順で検証すること。検証なしに指摘をそのまま適用してはならない。**

1. **該当スキルを確認する**: 指摘内容に関連するスキル（`test-writing`・`development-guidelines` 等）のガイドラインを参照し、指摘がそのガイドラインと一致しているか確認する
2. **既存コードを確認する**: プロジェクト内の類似実装（同じパターンの既存ファイル）を参照し、指摘が既存のコードパターンと一致しているか確認する
3. **判断する**:
   - スキルのガイドラインや既存コードと **一致する** 指摘 → 修正する
   - スキルのガイドラインや既存コードと **矛盾する** 指摘 → **スキルのガイドラインを優先し、修正しない**

> **原則**: コードレビューの指摘よりも、スキルのガイドラインと既存コードのパターンが常に優先される。レビュアーの指摘は参考情報であり、スキルの定義が唯一の基準である。
```

### 注意事項

- 問題がない観点の「改善が必要な点」セクションは省略する
- 観点Dは最重要のため、問題があれば必ず具体的なドキュメントの記述と実装コードを対比して説明する
- 親エージェントへの返答は整形済みのテキストのみとし、余計な前置きは省く
- ファイルパスには行番号を含め、ユーザーがすぐに該当箇所を特定できるようにする
